# HealthX WebDAV 同步方案设计

目标：把当前的「WebDAV 备份/恢复」换成**多设备双向同步**，并从结构上防止**误删除**与**误覆盖**。
参考了番茄阅读器 `WEBDAV_SYNC.md` 的思路，但针对血压数据做了更强的处理。

---

## 0. 为什么能做得比「书架 LWW 整体替换」更稳

番茄书架用的是 **Last-Write-Wins 整体替换 + 无墓碑**，它自己承认两个缺陷：
- 两设备各改各的，较新一方会**整体抹掉**较旧一方的改动；
- 删除靠整体替换传播，**分不清「对方删了」和「对方还没同步到」**。

血压记录的特点让我们能避开这两个坑：
- 每条记录是**独立事件**、有唯一身份、几乎不被修改；
- 因此可以做**按记录合并（并集）**，而不是整份文件谁新谁赢。

结论：采用 **LWW-Element-Set（按记录的增/删集合）** —— 每条记录带自己的 `updatedAt` 和 `deleted` 墓碑，合并时按记录取新。并集合并 + 墓碑，天然满足「不误删、不误覆盖」。

---

## 1. 数据模型改动

给 `bp_record` 增加三列（Room 迁移 v1 → v2）：

| 列 | 类型 | 含义 |
|---|---|---|
| `uid` | TEXT | **跨设备唯一 ID**（UUID）。同步以它为主键，本地自增 `id` 仅用于本机/导航 |
| `updatedAt` | INTEGER | 该记录最后一次变更（新增/编辑/删除）的时间(ms)，冲突时取新 |
| `deleted` | INTEGER | 墓碑标记：0=存在，1=已删除 |

- **删除改为软删除**：`deleted=1` 且 `updatedAt=now`，不再物理删行。界面查询一律 `WHERE deleted=0`。
- 迁移时给存量记录补 `uid = lower(hex(randomblob(16)))`、`updatedAt = timestamp`、`deleted=0`，并对 `uid` 建唯一索引。

> 为什么必须加 uid：本地自增 `id` 在不同设备会**撞号**（两台设备都有 id=1 的不同记录）。用 UUID 才能跨设备正确对齐同一条记录。

同步文件（WebDAV 上，固定名 `healthx-sync.json`）：

```json
{
  "app": "HealthX",
  "schema": 2,
  "updatedAt": 1751500000000,      // 文件级最后写入时间（辅助/展示用）
  "records": [
    { "uid": "a1b2...", "timestamp": 1751490000000, "systolic": 120,
      "diastolic": 80, "heartRate": 72, "status": "normal", "note": "",
      "updatedAt": 1751490000000, "deleted": false }
  ]
}
```
墓碑也在 `records` 里（`deleted:true`），这样删除才能传播。

## 2. 同步算法（核心）

`SyncManager.sync()`，单线程后台执行：

```
1. PROPFIND：连通性 + 文件是否存在 + 记录远端 ETag（若服务器支持）
2. 若远端文件存在 → GET 下载并解析 SyncPayload；解析失败 → 中止（绝不用本地覆盖读不懂的文件）
3. 读取本地全部记录（含墓碑）
4. 合并（按 uid 做并集）：
     - 只在本地   → 保留本地
     - 只在远端   → 保留远端
     - 两边都有   → 取 updatedAt 更大的那条（含 deleted 状态）；相等则保留远端（幂等）
5. 把合并结果写回本地：按 uid upsert；远端更新的覆盖本地、远端墓碑标记本地删除、本地独有的保留待上传
6. 上传前快照：把「当前远端文件内容」另存为 history/healthx-sync-<ts>.json，仅保留最近 K 份（默认 10）
7. 条件上传：PUT 合并结果，带 If-Match: <远端ETag>（或 If-Unmodified-Since）
     - 200/201 成功 → 记录 lastSyncAt 与新 ETag
     - 412（远端在我们读取后被别的设备改过）→ 回到第 2 步重新拉取合并重试（最多 N 次）
```

要点：
- **并集合并** ⇒ 上传永远不会丢掉「只在服务器上存在」的记录（除非本地有更新的墓碑），**旧设备无法覆盖掉新数据**。
- **墓碑** ⇒ 删除能双向传播，且能区分「已删除」与「尚未同步到」。
- **条件 PUT（ETag）** ⇒ 两设备几乎同时同步时，防止后写者盖掉先写者（丢更新）。服务器不支持 ETag 时自动降级（见下）。

## 3. 你关心的三类场景，如何保证安全

| 场景 | 结果 | 为什么安全 |
|---|---|---|
| **新设备接入已有 WebDAV 备份** | 拉全远端记录到本地，本地空不会删服务器任何东西 | 本地无墓碑，合并= 远端 ∪ ∅ = 远端。**空本地永远删不掉远端记录** |
| **旧设备接入全新/空 WebDAV** | 远端空 → 上传本地全部，服务器补齐 | 合并= ∅ ∪ 本地 = 本地，无丢失 |
| **多设备并发各加各的** | 两边的新记录都在（并集），互不覆盖 | 按 uid 合并，不是整份 LWW |
| **某设备编辑同一条** | 取 `updatedAt` 新的版本 | 记录级 LWW，仅影响那一条 |
| **误删一批记录** | 删除会传播，但服务器留有历史快照可回滚 | 第 6 步的 history 快照 |
| **两设备同时同步** | 后写者遇 412 重新合并再传 | 条件 PUT 乐观锁 |

额外「防手滑」阀门：
- ✅ **服务器历史快照（已选用）**：每次上传前把当前远端文件另存为 `history/healthx-sync-<ts>.json`，仅保留最近 10 份，误删后可手动回滚。这是本方案采用的防误删保险。
- ✅ **fresh-install 强信任服务器**：本地从未同步且为空、而远端有数据时，先下载后上传（并集本已保证，此项是双保险）。
- ⛔ **危险删除阈值确认（本次不做）**：曾考虑「净删除超过阈值就暂停并弹窗确认」，本次不启用；靠上面的历史快照回滚即可。

## 4. 触发时机

沿用番茄的自动策略，并加一个手动入口：
- **App 回到前台**、**App 退到后台**（`ProcessLifecycleOwner`，全应用级，旋转屏幕不误触发）。
- 设置页「立即同步」按钮（手动）。
- 关闭同步开关则完全不触发。

---

## 5. ETag 不支持时的降级

坚果云等支持 `ETag`/`If-Match`；个别 WebDAV 不支持条件请求。降级顺序：
1. 优先 `If-Match: <ETag>`；
2. 无 ETag → 用 PROPFIND 的 `getlastmodified` + `If-Unmodified-Since`；
3. 都不支持 → 「下载→合并→上传」窗口尽量短，靠并集合并把竞态影响降到「最多某条记录以较早版本落地，下次同步自愈」，绝不整表丢失。

## 6. 保证 / 不保证

### ✅ 保证
- 空本地不会覆盖/删除服务器数据（新设备安全接入）。
- 旧设备接入空服务器不丢数据。
- 并发新增走并集，不互相覆盖。
- 删除有墓碑、可双向传播且可区分未同步。
- 误删有服务器历史快照可回滚。
- 同时同步有条件写入防丢更新。

### ⚠️ 不保证 / 取舍
- 记录级冲突仍是 LWW（同一条被两处编辑，较新覆盖较旧）——对血压数据可接受。
- 依赖设备时钟大致正确；某设备时钟严重超前可能让其版本总是「较新」。可后续引入逻辑时钟，暂不做。
- 墓碑会永久留存（表会缓慢增长）。可加「清理 N 天前墓碑」的维护任务，暂不做。

## 7. 实现改动清单

1. `BpRecord`：加 `uid/updatedAt/deleted`；`@Serializable` DTO 复用。
2. `BpDatabase`：version 2 + `Migration(1,2)`（加列、补 uid、建唯一索引）。
3. `BpDao`：显示类查询加 `deleted=0`；新增 `getAllIncludingDeleted()`、`upsertByUid`、`softDelete`。
4. `BpRepository`：`add` 生成 uid+updatedAt；`delete` 改软删除；新增合并写入方法。
5. 新增 `SyncManager`（替代 `BackupManager`）：下载→合并→快照→条件上传→重试。
6. `WebDavClient`：补 PROPFIND 取 ETag、条件 PUT（If-Match）、history 快照的复制/清理。
7. `HealthXApp`：前后台触发自动同步。
8. UI：把「WebDAV 备份 / 数据恢复」两个入口合并为「**WebDAV 同步**」一个页面（配置 + 开关 + 立即同步 + 上次同步时间 + 状态/冲突提示）。
9. 设置页文案相应调整。
